# 海龟汤游戏设计

## 编程语言

- HTML
- CSS
- Javascript

## 编码输出输入格式

utf-8

## 文本语言

- zh-CN
- en-US

## 要求

- 用户输入Deep Seek API Key用于连接LLM服务
- 用户可以选择本格推理或者非本格推理`is_traditional`
- 用户可以选择故事风格`story_style` （多选，或者允许玩家手动输入，默认为“悬疑推理”）
  - 惊悚；恐怖
  - 喜剧
  - 鬼怪玄幻
  - 刑侦悬疑
  - 仙侠；爽文
  - ...
- 让用户选择游戏难度`(enum)difficulty`
- 需要设计一套泛用系统提示词，让LLM根据系统提示词来编写相应的故事大纲和设计谜题。

## 难度设计

- 自定义模式`custom_model`
  - 用户可以自己定义题目难度`[difficulty.newb,difficulty.easy,difficulty.hard,difficulty.hardcore]`
  - 用户可以自己定义提问次数（至少5次，至多30次）
  - 用户可以自定义题目文本长度（200-10000字以内）

- 入门难度`difficulty.newb`
  - 面向完全新手玩家，对海龟汤游戏完全没有经验，在文本中适当使用富文本格式提示玩家，比如`<em>这是一段线索...</em>`
  - 题目难度入门，不需要非常丰富的想象能力也能还原故事大纲。
  - 对于故事细节部分，能够在1-3次提问中推断出来。
  - 用户提问次数为30次
  - 题目文本长度不因该超过500字。

- 简单难度`difficulty.easy`
  - 面向有一定游戏经验的新手玩家，不需要过多解释规则或者刻意暴露盘问点。
  - 题目难度简单，需要一定的生活常识/想象力，但不需要专业知识来推断。目标群体是绝大多数普通人，他们不会思考非常复杂的情况。
  - 盘问点需要设计一定的深度，玩家通常需要在2次以上的提问中才能推理出来细节。
  - 用户提问次数为20-30次。
  - 题目文本长度不低于300字，不应该超过1000字。

- 专家难度`difficulty.hard`
  - 面向经验丰富的玩家，可以暴露最多5个盘问点。
  - 题目难度适中，需要掌握生活常识，想象力丰富，根据`is_traditional`变量来决定是否需要一定的专业知识储备。目标群体是初中及以上学历的知识群体，他们应当掌握基础的义务教育知识。
  - 盘问点可以深化设计，但是不能超出义务教育知识水平。不论是本格推理还是非本格推理，应该控制盘问点难度在3次以上的提问。
  - 用户提问次数为10-20次。
  - 题目文本长度不少于300字，不超过2000字。

- 硬核难度`difficulty.hardcore`
  - 面向专业玩家，不允许使用富文本等手段暴露盘问点，可以通过颜色提示玩家“人物”/“角色”/“对象”等单位是同一个事物。
  - 题目难度为困难，玩家需要有丰富的生活知识以及专业知识。对于非本格推理可以尽可能多的省略关键信息，让玩家自行想象发挥。
  - 盘问点设计不再有难度上限限制，需要尽可能让玩家花费多次提问才能还原细节。
  - 用户提问次数为10-15次。
  - 允许在题目文本中添加干扰文本，诱导玩家，但内容不能偏离故事大纲。
  - 不允许添加无意义的文本。如果是重复文本，该文本必须提供重要的潜在信息（即必须是一个盘问点）。
  - 题目文本长度不少于200字，不超过10000字。

## 特殊规则

- 通常来说，LLM对于用户的提问仅需回复“是”/“不是”/“重要”/“不重要”/“是也不是”等简短回复。但对于特殊情况，LLM需要对用户的提问进行进一步解释说明。
- 用户的每次提问可能不会遵守规则，因此LLM需要控制回答内容，不能让用户作弊。

    ```example
    "user":"线索A是不是XXXX？线索B是不是XXX？"
    "LLM":"你不能连续提问多个问题。但我可以回答你第一个问题，答案是：是。"
    ```

- 用户在游玩过程中可能会陷入逻辑闭环，LLM需要及时引导玩家不再钻牛角尖。比如玩家已经超过2次盘问某个逻辑点，但该细节对还原海龟汤的贡献较低，甚至没有帮助时，LLM应当告诉玩家盘问这个逻辑点不是一个好选择。

  ```example
    "user":"线索xx是不是XXX？"
    "LLM":"线索xx不重要。"
    ```

- 对于`difficulty.hardcore`难度，不需要引导玩家，玩家应当有判断力是否在钻牛角尖。LLM采取缄默法则，对玩家仅进行简短回复。

## 游戏流程

- 玩家在配置完基础信息以后，系统需要根据玩家的设置来配置相应的LLM工程提示词。
- 工程提示词需要让LLM返回故事大纲（string）、谜面（string）、盘问点（string[]）以及汤底（string）。
- 设置一个相应个盘问点计数器，每当玩家成功发现盘问点，并获取了正确信息以后，盘问点计数器更新。
- 对于谜面，需要在系统提示词中让LLM标注相应的富文本标签（直接返回HTML语言最佳）。
- 在收到LLM返回的谜面以后，将谜面渲染到网页中。
- 需要配置UI提示，让玩家输入问题，并查看LLM返回的回答。
- 需要配置提问剩余次数，让玩家能够直观地看到还剩多少提问机会。
- 需要配置UI来告诉玩家当前故事推理进度（盘问点更新进度）。
- 在所有盘问点被推理出来以后，解锁汤底提交按钮，提前公布游戏结果。
- 在所有提问机会被用完以后，强制玩家提交汤底，并公布游戏结果。
- 根据玩家的汤底，返回故事大纲吻合度，并让LLM打分（100分制，受提问次数和汤底准确度影响）。并让LLM生成一份报告，告诉玩家失误点和提升技巧。
- 在游戏结束以后提供导出json到本地功能，将故事大纲（string）、谜面（string）、盘问点（string[]）以及汤底（string）以及玩家提问、行为，以及LLM的思考过程存入到json中以便复盘。
